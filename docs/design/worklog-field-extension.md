## 扩展字段 (建议新增)

### P0 — 立即实施（高价值、可自动填充）

#### 1. 标签（多选）

**选项**：#开发 #文档 #学习 #会议 #排查 #部署 #沟通 #管理

**自动填充逻辑**：从"成果类型"映射
- 工具开发 → #开发
- 文档输出 → #文档
- 学习笔记 → #学习
- 问题排查 → #排查
- 技术方案 → #部署
- 项目交付 → #管理

**价值**：支持跨 KR 按工作性质筛选（如"本周写了多少文档"），弥补纯 KR 维度的不足。

**实施**：SessionEnd hook `sync_to_feishu()` 中自动设标签；存量 443 条用脚本批量回填。

#### 2. 状态（单选）

**选项**：完成 / 进行中 / 受阻 / 已取消

**自动填充逻辑**：SessionEnd 自动生成的 → 默认"完成"。手动补记的 → 默认"进行中"。

**价值**：区分"已完成的碎片 session"和"真正未完成的工作项"。

**实施**：hook 写 worklog 时设"完成"；log_write.py 手动补记时可选。

### P1 — 下个迭代（高价值、需少量开发）

#### 3. 耗时(分钟)（数字）

**估算逻辑**：asst_msgs × 5 分钟（粗略估计每轮对话 ≈ 5 分钟实际工作时间）

**价值**：
- 周报/季报：本周投入 XX 小时在 ccconfig，YY 小时在 bwater
- OKR 进度：计划 vs 实际时间分配对比
- 识别时间碎片化：每天 > 3 个 session → 上下文切换过多

**实施**：hook 自动写入估算耗时；提供 log_write.py 手动修正接口。

### P2 — 有需求时做（手动为主、自动为辅）

#### 4. 阻碍项（文本）

**自动检测**：user prompt 中含"报错"/"卡住"/"不行"/"问题"且 asst_msgs > 5（讨论很久）→ 标记。

**价值**：月度/季度总结时汇总阻碍模式，找系统性问题。

**实施**：下个迭代。先用手动补记方式验证需求。

#### 5. 下一步（文本）

**完全手动**：session 结束时用户说"记录下一步：XXX"→ 写入。

**价值**：跨 session 连续性的关键。但目前 SessionEnd 没有交互能力。

**实施**：远期。考虑 hook 结束后提示用户输入下一步（需要 Claude Code 支持 hook 回调）。

## 实施步骤

### Step 1：飞书 Base 加字段（手动操作）

在 Worklog 表 (`tblVsC0L7QFzMeYM`) 添加：

- 标签从成果类型映射
- 状态默认"完成"

### Step 3：存量数据回填

- 标签：按成果类型批量映射
- 状态：全设"完成"（历史数据）
- 耗时：asst_msgs × 5

### Step 4：更新 data-model.md

同步文档与实际表结构。

## 不做的

| 字段 | 原因 |
|------|------|
| 心情/评分 | 个人偏好，不适合自动化 |
| 附件 | Base 附件字段 API 复杂，当前无需求 |
| 关联O（直连） | 已通过 KR→O 间接关联，无需冗余 |
| 开始时间 | 当前"日期"字段足够（日粒度），小时粒度无价值 |
