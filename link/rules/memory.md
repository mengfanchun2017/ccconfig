# 记忆与提交规范

## 自动记忆
Claude 应在以下情况自动保存 memory，无需用户提醒：
- 完成任何脚本/配置的创建或重写后
- 学习了新的工作流或操作步骤后
- 发现并修复了 bug 后
- 架构/配置发生变更后
- 技术调研得出明确结论/决策后（如工具选型、方案对比、能力验证）

**Why**: 用户不希望每次都说"记住这个"或"写到 memory"。
**How to apply**: 在完成上述任务后，自动判断是否需要保存 memory 和更新 MEMORY.md 索引，不要等用户提醒。

## 变更摘要
每次修改任何文件后，必须在回应末尾输出变更摘要表格，列出所有改动文件及变更内容。

**Why**: 用户需要清晰了解每次会话改了什么，不让变更静默发生。
**How to apply**:
- 汇总所有改动过的文件：脚本、agent、memory、配置、规则等
- 格式：`| 文件 | 变更 |`，变更类型：新增/修改/删除 + 一句话说明
- 不止 memory 文件，所有文件改动都要列出

## 试错记录
命令行工具、API 等通过试错（先失败再成功）获得的正确用法，必须立即记录到对应的速查文件或 memory，避免后续会话重蹈覆辙。
- 命令失败 → 调整参数 → 成功 → 立即更新对应 rule 或 skill 的速查表
- 新工具新建 memory；已有工具追加到速查表
- 记录格式：❌错误写法 → ✅正确写法 + 失败原因

**Why**: 每次新会话没有之前的试错记忆，会重复走弯路。记录后将隐式知识显式化，跨会话复用。

## 主动提交
在 git 仓库中编辑/写入文件后，必须在当轮或下一轮主动 `git add` + `git commit`，不等用户说"提交"。

**Why**: 2026-05-23 会话中修改了 init.sh、gitforce.sh、monitor.sh 三个文件都没 commit，用户运行 gitforce.sh 发现大量未提交改动。用户明确说"不应该你修改文件之后就提交么"。
**How to apply**:
- 任何 Edit/Write 操作后，检查文件是否在 git 仓库中
- 如果是，在当轮对话结束前或下一轮开始时主动 commit
- commit message 用中文描述 why
- ccconfig 下提交格式: `Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>`
